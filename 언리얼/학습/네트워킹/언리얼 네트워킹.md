# 개요
이 문서는 언리얼 엔진 네트워킹을 한 번에 훑는 통합 학습 노트다.
핵심 목표는 `서버 권한`, `역할(Role)`, `소유(Owner)`, `복제(Replication)`, `RPC`가 어떻게 연결되는지 하나의 흐름으로 이해하는 것이다.

언리얼 엔진의 멀티플레이는 기본적으로 `server-authoritative client-server model`이다.
즉 서버가 진실을 가지며, 클라이언트는 그 상태를 복제받고, 필요한 요청만 서버에 올린다.
이 문서는 공식문서의 네트워킹 핵심 페이지들을 기반으로, 엔진 학습 관점에서 전체 구조를 정리한다.

## 전제조건
- 멀티플레이의 진실은 서버가 가진다.
- 클라이언트는 서버 상태의 복사본을 본다.
- 입력, 요청, 의도는 서버로 올리고, 결과 상태는 다시 서버가 확정한다.
- 액터, 컴포넌트, 서브오브젝트의 네트워크 동작은 ownership과 relevancy의 영향을 받는다.

# 가장 먼저 잡아야 할 네 가지

언리얼 네트워킹을 볼 때는 아래 네 가지를 항상 같이 봐야 한다.

| 개념 | 의미 | 왜 중요한가 |
| --- | --- | --- |
| `Authority` | 누가 진짜 상태를 결정하는가 | 상태 변경의 출발점 |
| `Role` | 로컬/원격 머신이 이 액터를 어떤 프록시로 보는가 | 시뮬레이션과 권한 구분 |
| `Ownership / Owning Connection` | 이 액터가 어느 플레이어 연결과 연결되는가 | RPC 대상 결정 |
| `Relevancy` | 이 액터를 특정 클라이언트에게 지금 보내야 하는가 | 실제 전송 여부 결정 |

# 권한과 역할

## Actor Role과 Remote Role
공식문서 기준으로 `Role`과 `RemoteRole`은 이 게임 인스턴스가 해당 액터를 얼마나 제어할 수 있는지를 보여준다.

대표 상태:

| 역할 | 설명 |
| --- | --- |
| `ROLE_None` | 복제되지 않음 |
| `ROLE_SimulatedProxy` | 상태를 시뮬레이션하지만 권한은 없음 |
| `ROLE_AutonomousProxy` | 시뮬레이션과 입력 권한이 있는 프록시 |
| `ROLE_Authority` | 진짜 상태를 가진 권한 측 |

공식문서가 특히 강조하는 점은 다음이다.

- `Role`은 `ownership`과 같은 개념이 아니다.
- 복제 액터는 서버에서만 `ROLE_Authority`를 갖는 것이 일반적이다.
- 클라이언트는 `SimulatedProxy` 또는 `AutonomousProxy`로 액터를 본다.

## `HasAuthority()`의 의미
실무적으로 `HasAuthority()`는 `이 위치가 상태를 확정할 수 있는가`를 보는 질문에 가깝다.
즉 서버에서만 진짜 게임 규칙을 바꾸는 습관이 중요하다.

> [!info]
> 공식문서도 role은 "이 액터를 이 머신이 얼마나 제어하는가"를 설명하는 기준이라고 본다. 따라서 `RPC 호출 가능 여부`, `복제 주체`, `시뮬레이션 방식`을 role과 같이 보는 습관이 중요하다.

# 소유(Owner)와 Owning Connection

## 왜 중요한가
공식문서 `Actor Owner and Owning Connection`은 ownership이 다음 세 영역에 중요하다고 설명한다.

- 액터 복제
- 프로퍼티 복제 조건
- RPC

즉 `Owner`와 `Owning Connection`은 RPC의 대상과 owner 기반 조건 복제의 기준이다.

## 기본 관계
일반적인 흐름은 아래와 같다.

- 클라이언트가 서버에 접속한다.
- 서버에 그 클라이언트용 `PlayerController`가 생성된다.
- 그 `PlayerController`가 `Pawn`을 소유한다.
- 그 `Pawn`의 owning connection은 그 플레이어의 연결이다.

같은 논리로 인벤토리 아이템, 소유 컴포넌트, 플레이어 소유 시스템도 같은 연결을 따를 수 있다.

## 대표 함수
- `AActor::GetOwner`
- `UActorComponent::GetOwner`
- `AActor::GetNetConnection`

> [!tip]
> 실무에서 `PlayerController`, possessed `Pawn`, 소유된 `ActorComponent`가 RPC 진입점으로 자주 쓰이는 이유는 이 ownership 체인이 분명하기 때문이다.

# Replication과 RPC의 차이

## 상태와 이벤트
언리얼 네트워크 문제는 크게 두 종류다.

- 현재 상태를 어떻게 동기화할 것인가
- 어떤 동작을 어떻게 전달할 것인가

이 둘을 언리얼에서는 보통 아래처럼 푼다.

| 문제 | 메커니즘 |
| --- | --- |
| 현재 상태 | `Replication` |
| 요청, 응답, 순간 이벤트 | `RPC` |

즉:
- 체력, 점수, 문 열림 여부 -> `Replication`
- 발사 요청, 상호작용 요청, 히트마커 -> `RPC`

# Replication 요약

## 액터 프로퍼티 복제
공식문서 기준 액터 프로퍼티 복제의 핵심은 아래다.

- `Replicated`
- `ReplicatedUsing`
- `GetLifetimeReplicatedProps()`
- `DOREPLIFETIME` 계열 매크로

대표 흐름:
1. 프로퍼티 선언
2. 액터 복제 활성화
3. 수명 복제 목록 등록
4. 필요 시 `RepNotify` 처리

## 조건부 복제
owner 전용, 초기 번들 전용, simulated 전용 등 연결별 필터링이 가능하다.
실무적으로는 `COND_OwnerOnly`, `COND_SkipOwner`, `COND_InitialOnly`를 가장 먼저 익히는 것이 좋다.

## 오브젝트 레퍼런스와 서브오브젝트
공식문서 기준 액터 외에도 `UObject`를 서브오브젝트로 복제할 수 있다.
대표 방식은 다음 두 가지다.

- `ReplicateSubobjects()`
- `AddReplicatedSubObject()`

현대적인 기준에서는 등록 리스트 방식이 더 중요하며, `Iris`와의 호환성도 이쪽이 좋다.

# RPC 요약

## RPC 종류
공식문서 기준 RPC 종류는 네 가지다.

| 타입 | 설명 |
| --- | --- |
| `Client` | owning client에서 실행 |
| `Server` | 서버에서 실행 |
| `Remote` | 연결의 반대편 원격에서 실행 |
| `NetMulticast` | 서버와 relevant한 클라이언트들에서 실행 |

## Reliability
- `Reliable`: 놓치면 안 되는 호출
- `Unreliable`: 유실 가능하지만 자주 오는 호출

## Send Policy
공식문서는 `Default`, `ForceSend`, `ForceQueue` 정책도 설명한다.
대부분은 `Default`로 충분하지만, 지연과 대역폭 제어를 의식하는 고급 단계에서는 의미가 있다.

## Validation
공식문서는 `Server RPC validation`을 통해 서버가 클라이언트 입력을 검증해야 한다고 설명한다.
즉 서버는 클라이언트를 신뢰하는 것이 아니라, 요청을 받고 규칙에 맞는지 확인한 뒤 실행한다.

# 네트워크 설계 순서

언리얼 네트워크 기능을 설계할 때는 아래 순서로 생각하면 가장 덜 꼬인다.

1. 이 정보가 `상태`인가 `이벤트`인가
2. 누가 진실을 결정하는가
3. 누가 이 정보를 봐야 하는가
4. owner 전용인가, 전체 공개인가
5. 지금 필요한 것은 `Replication`인가 `RPC`인가
6. late joiner도 알아야 하는가

이 질문에 답이 정해지면 수단도 거의 정해진다.

| 상황 | 선택 |
| --- | --- |
| 현재 값을 남겨야 함 | `Replication` |
| 클라이언트가 서버에 요청 | `Server RPC` |
| 서버가 특정 owner에게 응답 | `Client RPC` |
| 모두에게 보이는 일회성 연출 | `NetMulticast RPC` |
| 나중에 들어온 플레이어도 알아야 함 | 다시 `Replication`으로 남겨야 함 |

# 자주 하는 실수

> [!caution]
> `Role`과 `Ownership`은 다른 개념이다. `ROLE_Authority`라고 해서 곧 owner라는 뜻은 아니다.

> [!caution]
> `NetMulticast`는 과거 호출 이력을 보존하지 않는다. 따라서 상태를 multicast만으로 유지하면 late joiner가 맞는 상태를 못 본다.

> [!caution]
> 복제가 안 되는 문제는 단순히 `Replicated` 선언 누락만이 아니라 relevancy, dormancy, ownership 문제일 수도 있다.

> [!caution]
> 클라이언트가 replicated 변수를 바꿔도 서버 상태는 바뀌지 않는다. 진짜 상태 변경은 서버가 해야 한다.

# 일반 팁

- 네트워크 설계는 항상 `요청`과 `결과 상태`를 분리해서 본다.
- owner 개념이 불분명한 월드 액터는 직접 RPC 진입점으로 쓰지 않는 편이 안전하다.
- 액터 역할, owner, relevancy를 같이 보면 왜 값이 안 가는지 훨씬 빨리 찾을 수 있다.
- 엔진 학습 단계에서는 `Actor -> PlayerController -> Pawn -> ActorComponent` 관계를 먼저 몸에 익히는 것이 좋다.

## 정리
언리얼 네트워킹을 한 줄로 줄이면 다음과 같다.

- 서버가 상태를 확정한다.
- 상태는 `Replication`으로 맞춘다.
- 요청과 응답, 순간 이벤트는 `RPC`로 전달한다.
- 이 모든 과정은 `Role`, `Owner`, `Owning Connection`, `Relevancy` 위에서 동작한다.

즉 앞으로 엔진 학습을 이어갈 때도, 먼저 `권한`, `소유`, `전송 조건`을 보고 그다음에 `Replication`과 `RPC`를 읽는 순서가 가장 안정적이다.
