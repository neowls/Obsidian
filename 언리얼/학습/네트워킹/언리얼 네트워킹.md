# 개요
이 문서는 언리얼 엔진 네트워킹의 기본 축을 하나로 묶는 통합 허브다.
핵심은 `서버 권한`, `Role`, `Owner / Owning Connection`, `Relevancy / Priority / Frequency / Dormancy`, `Replication`, `RPC`가 어떻게 연결되는지를 엔진 코드 기준으로 잡는 것이다.

언리얼 멀티플레이는 기본적으로 `server-authoritative client-server model`이다.
즉 서버가 진실을 가지며, 클라이언트는 그 상태를 복제받고, 필요한 요청만 서버에 올린다.

## 전제조건
- 멀티플레이 상태의 진실은 서버가 가진다.
- 클라이언트는 서버 상태의 복사본을 본다.
- 입력, 요청, 의도는 서버로 올리고, 결과 상태는 다시 서버가 확정한다.
- 액터, 컴포넌트, 서브오브젝트의 네트워크 동작은 ownership과 relevancy의 영향을 받는다.

# 네트워킹을 볼 때 항상 같이 봐야 할 축

| 축 | 핵심 질문 | 대표 함수/변수 |
| --- | --- | --- |
| `Authority / Role` | 누가 상태를 확정하는가 | `HasAuthority()`, `GetLocalRole()` |
| `Ownership / Connection` | 이 액터는 어느 플레이어 연결에 속하는가 | `GetOwner()`, `GetNetOwner()`, `GetNetConnection()` |
| `Relevancy / Priority / Frequency / Dormancy` | 지금 이 connection에 보내야 하는가 | `IsNetRelevantFor()`, `GetNetPriority()`, `NetUpdateFrequency`, `NetDormancy` |
| `Replication` | 현재 상태를 어떻게 맞출 것인가 | `GetLifetimeReplicatedProps()`, `FRepLayout`, `OnRep_*` |
| `RPC` | 원격 함수 실행을 어떻게 보낼 것인가 | `GetFunctionCallspace()`, `CallRemoteFunction()` |

> [!info]
> `Authority`, `Owner`, `NetOwner`, `OwningPlayer`, `NetConnection`은 같은 말이 아니다. 이걸 한 덩어리로 보면 RPC와 복제 문제를 계속 헷갈리게 된다.

# 권한과 역할

## `HasAuthority()`의 의미
엔진 코드 기준으로 `HasAuthority()`는 결국 `GetLocalRole() == ROLE_Authority`다.
즉 질문은 `내가 이 액터의 진짜 상태를 확정할 수 있는가`이지, `내가 이 액터를 조종하는가`가 아니다.

## 대표 Role

| 역할 | 설명 |
| --- | --- |
| `ROLE_None` | 복제되지 않음 |
| `ROLE_SimulatedProxy` | 상태를 시뮬레이션하지만 권한은 없음 |
| `ROLE_AutonomousProxy` | 입력 기반 예측이 가능한 소유 클라이언트 프록시 |
| `ROLE_Authority` | 진짜 상태를 가진 권한 측 |

## 실무 해석
- 서버의 replicated actor는 보통 `ROLE_Authority`
- 소유 클라이언트의 캐릭터는 보통 `ROLE_AutonomousProxy`
- 다른 클라이언트가 보는 캐릭터는 보통 `ROLE_SimulatedProxy`

> [!caution]
> `ROLE_Authority`라고 해서 owner라는 뜻은 아니다. authority는 상태 확정 권한이고, owner는 연결 귀속 정보다.

# 소유(Owner)와 Owning Connection

## 왜 중요한가
ownership은 다음 세 가지에 직접 영향을 준다.

- `Client RPC` 대상 결정
- owner 기반 property 조건 복제
- actor/component/subobject의 네트워크 문맥

## 핵심 함수

| 함수 | 의미 |
| --- | --- |
| `GetOwner()` | 일반 actor owner 체인 |
| `GetNetOwner()` | replication/RPC 문맥의 책임 actor |
| `GetNetOwningPlayer()` | owning `UPlayer` |
| `GetNetConnection()` | 실제 `UNetConnection` |
| `UpdateOwningNetConnection()` | connection 변화를 replication system에 반영 |

## `PlayerController`와 `Pawn`이 특별한 이유
- `APlayerController`는 자기 자신이 net owner이고, player/connection을 직접 가진다.
- `APawn`은 possession 시점에 `SetOwner(NewController)`, `UpdateOwningNetConnection()`, `SetAutonomousProxy(true)`가 같이 움직인다.
- 그래서 실무에서 `PlayerController`, possessed `Pawn`, 그들의 replicated component가 RPC 진입점으로 자주 쓰인다.

## 중요한 구분
- 기본 actor는 `GetNetConnection()`을 owner chain에 위임한다.
- `Pawn`은 `GetNetOwner()`는 자기 자신을 반환할 수 있지만, 실제 `OwningPlayer`와 `NetConnection`은 controller를 따라간다.
- `UActorComponent`의 RPC 판단도 결국 owner actor 문맥을 그대로 탄다.

> [!tip]
> 월드에 그냥 놓인 일반 actor에 클라이언트가 직접 `Server RPC`를 거는 방식은 ownership 때문에 자주 막힌다. 보통은 플레이어가 소유한 actor를 통해 서버에 요청하는 흐름이 더 안전하다.

# Relevancy / Priority / Frequency / Dormancy

## 왜 이 층이 필요한가
replicated property가 등록되어 있어도, 언리얼은 모든 actor를 모든 connection에 매 프레임 보내지 않는다.
이 단계는 `이번 프레임에 이 actor를 이 connection에 보낼지`를 결정하는 스케줄링 계층이다.

## 네 가지 질문

| 개념 | 질문 | 대표 API |
| --- | --- | --- |
| `Relevancy` | 지금 이 connection에 보여야 하는가 | `AActor::IsNetRelevantFor()` |
| `Priority` | relevant하다면 무엇을 먼저 보낼까 | `AActor::GetNetPriority()` |
| `Frequency` | 다시 검사할 최소 시점은 언제인가 | `NetUpdateFrequency`, `MinNetUpdateFrequency` |
| `Dormancy` | 한동안 검사 대상에서 빼도 되는가 | `NetDormancy`, `FlushNetDormancy()`, `ForceNetUpdate()` |

## `IsNetRelevantFor()`가 보는 것
핵심 early-out은 아래 순서로 이해하면 된다.

1. `bAlwaysRelevant`
2. `IsOwnedBy(ViewTarget)` / `IsOwnedBy(RealViewer)`
3. `bNetUseOwnerRelevancy`면 owner에게 위임
4. `bOnlyRelevantToOwner`면 owner가 아닌 connection에는 false
5. attach parent relevancy
6. hidden + no collision
7. 거리 기반 cull (`NetCullDistanceSquared`)

즉 relevancy는 단순 거리 계산이 아니라 owner, view target, attach 상태까지 같이 본다.

## `GetNetPriority()`와 starvation 방지
`GetNetPriority()`는 결국 `NetPriority * Time` 구조다.
즉 오래 못 보낸 actor는 우선순위가 점점 올라간다.

## `NetUpdateFrequency`
이 값은 `전송 빈도`보다 `consider list에 다시 들어올 최소 시점`에 가깝다.
즉 값이 안 가는 문제를 전부 frequency 탓으로 보면 오답이 많다.

## `Dormancy`
- `DORM_Awake`: 활성
- `DORM_DormantAll`: 모든 relevant connection에서 휴면 가능
- `DORM_DormantPartial`: viewer별 dormant 여부를 override로 결정
- `DORM_Initial`: 초기 전송 후 휴면에 가까운 특수 상태

`ForceNetUpdate()`는 `NextUpdateTime`을 당기고, dormant actor면 `FlushNetDormancy()`까지 호출한다.

> [!caution]
> 값이 안 간다고 해서 항상 등록 문제는 아니다. `owner`, `relevancy`, `frequency`, `dormancy`, `channel saturation` 중 어디서 막히는지 분리해서 봐야 한다.

# Replication과 RPC의 역할 분리

## 상태와 이벤트

| 문제 | 메커니즘 |
| --- | --- |
| 현재 상태를 맞춘다 | `Replication` |
| 요청, 응답, 순간 이벤트를 보낸다 | `RPC` |

예:
- 체력, 점수, 문 열림 상태 -> `Replication`
- 발사 요청, 상호작용 요청, 개인 HUD 응답 -> `RPC`

## 설계 순서
1. 이 정보가 `상태`인가 `이벤트`인가
2. 누가 진실을 결정하는가
3. 누가 이 정보를 봐야 하는가
4. owner 전용인가, 전체 공개인가
5. late joiner도 알아야 하는가

## 자주 하는 실수

> [!caution]
> `NetMulticast`는 과거 호출 이력을 late joiner에게 재생하지 않는다. 영속적 결과는 결국 replicated state로 남겨야 한다.

> [!caution]
> 클라이언트가 replicated 변수를 바꿔도 서버 상태는 바뀌지 않는다. 진짜 상태 변경은 서버가 해야 한다.

> [!caution]
> `Reliable`은 안전 옵션이 아니라 대역폭과 큐를 더 쓰는 옵션이다. 습관적으로 붙이면 안 된다.

## 문서 간 관계
- `언리얼 네트워킹`은 기본 개념 허브다.
- `Replication`은 registration -> filtering -> receive -> repnotify까지 포함한 상태 복제 문서다.
- `RPC`는 종류/신뢰성/validation뿐 아니라 callspace -> send -> receive 내부 경로까지 포함한다.
- `Fast Array`와 `CharacterMovement`는 일반 복제 규칙 위에서 동작하는 별도 심화 주제다.

# 문제를 볼 때의 디버깅 순서

1. 이 actor/component가 실제로 replicated인가
2. 현재 머신에 authority가 있는가
3. owner / net connection이 실제로 있는가
4. relevancy / dormancy / frequency에서 걸리고 있지 않은가
5. 이 문제는 상태 동기화 문제인가, 원격 함수 호출 문제인가

> [!tip]
> 네트워크 설계는 항상 `요청`과 `결과 상태`를 분리해서 보면 구조가 훨씬 덜 꼬인다.

## 정리
언리얼 네트워킹을 한 줄로 줄이면 다음과 같다.

- 서버가 상태를 확정한다.
- 상태는 `Replication`으로 맞춘다.
- 요청과 응답, 순간 이벤트는 `RPC`로 전달한다.
- 이 모든 과정은 `Role`, `Owner`, `Owning Connection`, `Relevancy` 위에서 동작한다.

즉 앞으로 엔진 학습을 이어갈 때도, 먼저 `권한`, `소유`, `전송 조건`을 보고 그다음 `Replication`과 `RPC`를 읽는 순서가 가장 안정적이다.
