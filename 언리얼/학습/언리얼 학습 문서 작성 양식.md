특히 이 세션에서는 `프로젝트 기능 문서`가 아니라 `언리얼 엔진 학습 문서`만 이 양식을 따른다.

## 핵심 원칙

- 가능하면 문서 첫 줄에 `언리얼 공식문서 링크`를 둔다.
- 제목은 `한글 + 필요 시 영어 원문 병기` 형식으로 쓴다.
- 첫 본문 섹션은 가능하면 항상 `개요`로 시작한다.
- 개념 설명은 먼저 쉬운 한국어로 풀고, 이후 원문 용어를 병기한다.
- 에디터 경로, 메뉴 경로, 패널 이름은 실제 표기대로 적는다.
- 속성, 옵션, 타입, 조건은 표로 정리한다.
- 주의사항, 팁, 예외는 Obsidian callout으로 정리한다.
- 이미지가 필요하면 `![[...]]` 형식의 Obsidian 임베드를 사용한다.
- 마지막에는 가능하면 `요약`, `일반 팁`, `엔진 소스 참고 포인트` 중 하나 이상을 둔다.

## 권장 문서 구조

공식문서 기반 학습 노트는 아래 순서를 기본으로 한다.

1. 공식문서 링크
2. `# 개요`
3. 핵심 개념 섹션
4. 세부 기능/옵션/동작 방식 섹션
5. 사용 절차 또는 워크플로
6. 주의사항 / 팁 / 예외
7. 디버깅 또는 활용 팁
8. 필요 시 엔진 소스 참고 포인트

## 문체 규칙

- 번역체를 유지하되, 너무 직역투로 끊기지 않게 다듬는다.
- 첫 문장은 정의, 두 번째 문장은 의미, 세 번째 문장은 실전 맥락으로 정리하는 식을 선호한다.
- 문단 중간에는 원문 용어를 괄호로 병기한다.
- 한 문단이 너무 길면 `##`, `###`로 쪼갠다.
- 기능 설명과 의견을 섞지 않는다. 의견은 `팁`, `주의`, `정리` 섹션으로 분리한다.

예시:
- `애니메이션 빠른 경로(Animation Fast Path)`
- `링크된 애님 레이어(Linked Anim Layers)`
- `프로퍼티 액세스(Property Access)`

## 표 작성 규칙

다음 항목은 가능하면 표로 정리한다.

- 프로퍼티 목록
- 옵션 목록
- 타입 비교
- 조건 비교
- 장단점 비교
- 클래스 역할 분리

예시:

| 종류 | 설명 | 언제 쓰는가 |
| --- | --- | --- |
| `COND_OwnerOnly` | owner에게만 보냄 | 개인 정보 복제 |
| `COND_SkipOwner` | owner 제외 전송 | owner는 이미 아는 값 |

## Callout 사용 규칙

- `> [!info]`: 공식문서 원문에서 중요 포인트
- `> [!tip]`: 실무 팁, 사용 요령
- `> [!caution]`: 오해하기 쉬운 제약, 성능 비용, 함정
- `> [!example]`: 이해를 돕는 예시
- `> [!note]`: 문맥상 덧붙일 설명

## 이미지 사용 규칙

- 공식문서 흐름을 그대로 따라갈 때는 이미지도 필요한 지점에 같이 둔다.
- 같은 설명이 글만으로 충분하면 이미지는 생략한다.
- 이미지 직전 문단에서는 "무엇을 봐야 하는지"를 먼저 적는다.
- 이미지 뒤에는 가능하면 한 줄 해설을 덧붙인다.

## 엔진 학습 문서에서 추가하면 좋은 섹션

엔진 개념을 깊게 파는 문서는 아래 섹션을 추가해도 좋다.

### 엔진 내부 관점
- 이 기능이 엔진에서 어떤 시스템 위에 얹혀 있는지
- 관련 클래스와 책임

### 실무 해석
- 프로젝트에서 이 기능을 언제 선택할지
- 비슷한 기능과의 구분

### 디버깅 체크리스트
- 왜 안 되는지 볼 순서
- 자주 놓치는 조건

### 엔진 소스 참고 포인트
- `Actor.cpp`
- `NetDriver.cpp`
- `RepLayout.cpp`
처럼 직접 읽을 가치가 있는 파일과 이유

## 앞으로의 적용 규칙

앞으로 내가 새로 작성하는 `언리얼\학습` 아래의 엔진 학습 문서는 아래를 기본값으로 삼는다.

- 공식문서 기반이면 첫 줄에 출처 링크를 남긴다.
- 섹션은 `개요 -> 핵심 개념 -> 세부 항목 -> 팁/주의 -> 소스 참고` 흐름을 우선한다.
- 한글 설명을 우선하고, 영어 원문 용어를 괄호로 병기한다.
- 표, callout, 이미지 배치를 적극적으로 사용한다.
- 필요하면 마지막에 `내 해석`보다 `엔진 소스 참고 포인트`를 우선 추가한다.

## 현재 세션에서의 해석

이 세션에서는 `NightCaretaker`를 현재 프로젝트 맥락으로 알고는 있되,
작성되는 학습 문서는 프로젝트 설계서가 아니라 `언리얼 엔진 학습 노트`로 작성한다.
즉 앞으로 생성하는 네트워킹, 빌드, 월드 파티션, PCG, 오디오 관련 문서도 이 양식을 따른다.
