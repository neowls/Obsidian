# UI 시스템 설계 구현
## Slate 와 UMG 차이점과 사용 상황
Slate 는 언리얼 엔진에서 제공하는 저수준 UI 프레임워크 이며, C++ 기반으로 동작하고 렌더링 및 로직 처리를 개발자 세밀하게 처리할 수 있는게 장점입니다.
반면에, UMG는 언리얼 모션 그래픽의 약자로 앞서 말한 Slate 를 기반으로 만들어진 고수준 UI 툴입니다.
그래서 언리얼 에디터 안에서 비주얼 방식으로 UI를 구성할 수 있게 해주어서 디자이너와의 협업이 쉽고 빠른게 장점입니다.
보통 성능이 중시된는 상황이라면 Slate 를 사용하고, 직관적인 디자인 뭐 인벤토리나 메뉴 HUD 등이 필요한 부분이 있다면 UMG 가 주로 사용됩니다.

## 위젯 간 데이터 전달 방식은 어떻게 하나요?
저는 보통 **Delegate**나 **TWeakPtr**을 활용해서 하위 위젯에서 부모 위젯에 이벤트를 전달하거나,부모에서 자식에게 데이터를 세팅하는 구조를 씁니다.
특히 하위 위젯이 많아질수록 직접 참조보다는 이벤트 방식으로 연결하는 게  의존성이 줄어들고 유지보수가 편하다고 느낍니다.

🗝️ **키워드**: `Delegate`, `TWeakPtr`, `Event Dispatch`, `SetItem()`, `Broadcast`

## 인벤토리 시스템을 UMG로 어떻게 설계하고, 아이템 드래그앤 드롭은 어떤 방식으로 처리했나요?
UMG에서는 `UDragDropOperation`을 상속한 클래스를 통해  드래그 상태를 정의하고, `OnDragDetected`, `OnDrop`, `OnMouseButtonDown` 이벤트로 아이템 이동이나 스왑을 처리했습니다. 
UI 구조는 아이템 슬롯 위젯과 그리드 패널로 나눠서  DataTable 기반 아이템 목록을 동적으로 로딩하고,  드래그할 땐 슬롯에서 아이템 데이터를 넘기고,  드롭 위치에서 그걸 다시 받아서 갱신하는 방식으로 구현했습니다.

🗝️ **키워드**: `UDragDropOperation`, `OnDrop`, `Slot`, `ItemData`, `DataTable`, `Blueprint Interface`

## Slate로 커스텀 위젯을 만들어본 경험이 있다면 어떤 구조로 개발했나요?
Slate에서는 `SCompoundWidget` 기반으로 UI를 구성했고, 플러그인 형태로 리스트뷰 + 디테일 패널 구조를 만든 경험이 있습니다. 
`SListView<TSharedPtr<DataType>>`로 항목 리스트를 생성하고, 선택 시 `SDetailPanel`에서 항목 세부정보를 수정할 수 있게 구성했습니다. 
각 위젯은 약한 참조(`TWeakPtr`)로 연결했고, `RequestListRefresh()`로 변경 사항을 반영했습니다.

🗝️ **키워드**: `SCompoundWidget`, `SListView`, `TSharedPtr`, `RequestListRefresh`, `SDockTab`, `Slate Plugin`

# 게임 콘텐츠 연동
## 퀘스트 UI에서 데이터를 어디서 어떻게 받아오나요?
보통 퀘스트 시스템에서 **UI용으로 가공된 구조체 또는 정보**를 노출하고, UI는 그걸 `GetQuestList()` 같은 함수로 받아옵니다. 
변경 사항이 생기면 `OnQuestUpdated` 같은 Delegate를 Broadcast해서 UI에서 `Bind`한 함수가 UI를 다시 갱신하게끔 구성합니다.

🗝️ **키워드**: `Delegate`, `QuestManager`, `OnQuestUpdated`, `BlueprintAssignable`, `Notify System`

## NPC와의 상호작용 후 UI 갱신은 어떤 방식으로 처리했나요?
NPC와 상호작용하면 퀘스트나 대화 정보가 바뀌는 경우가 많아서,  대부분은 상호작용 끝에 해당 정보를 `QuestUI` 쪽으로 넘기거나  `UIController` 같은 중간 매니저가 `RefreshUI()`를 호출하는 방식으로 썼습니다. 
UI는 데이터를 Get하는 방식보다, 이벤트로 푸시 받는 형태가 깔끔하다고 느껴요.

🗝️ **키워드**: `NPCInteraction`, `UIController`, `Push vs Pull`, `OnInteractionComplete`

## UI에서 이벤트 트리거나 콜백은 어떻게 구성하나요?
UI 쪽에서 콜백이 필요한 경우, 주로 `BlueprintAssignable` 델리게이트를 선언해서  
다른 블루프린트나 C++에서 바인딩하도록 구성합니다.  
또는 Slate에서는 `FSimpleDelegate`, `FOnClicked`, `FOnTextCommitted` 같은 방식으로  
함수를 전달해서 로직을 분리했습니다.

🗝️ **키워드**: `BlueprintAssignable`, `FSimpleDelegate`, `Event Binding`, `UMG -> Game Logic 연동`

# UX 개선 및 최적화
## UI 성능 병목을 어떻게 확인하나요?
`stat GPU`, `stat Slate` 명령어로 각 위젯이 차지하는 비용을 확인하거나, `Unreal Insights`로 Slate 렌더 타임과 디버깅 이벤트를 추적합니다.  
특히 잦은 바인딩이나 `Tick` 발생이 많은 구조에서 병목이 발생하기 쉬워서 `RetainerBox`나 `InvalidationBox`를 적극 활용합니다.

🗝️ **키워드**: `stat Slate`, `Unreal Insights`, `RetainerBox`, `Invalidate Widget`, `Rebuild Widget Tree`

## UI Tick 관련 최적화 경험이 있나요?
데이터 바인딩을 많이 쓰면 매 프레임 `Tick`이 발생해서  큰 UI에서는 성능 이슈가 생깁니다.  
그래서 반복적으로 갱신되는 데이터는 **OnChanged 이벤트 기반 갱신**으로 바꾸거나, `CanTick`을 `false`로 설정해서 불필요한 업데이트를 줄인 적 있습니다.

🗝️ **키워드**: `Tick Frequency`, `Bindings vs Manual Refresh`, `CanTick`, `Performance Tips`

## Slate 에서 더 나은 성능을 위해 어떤 설계를 했나요?
Slate는 명시적으로 갱신하지 않으면 계속 렌더링 트리를 재구성하기 때문에,  고정된 위젯은 `InvalidationPanel` 또는 `Invalidate()` 호출로 최적화했고,  불필요한 `Slot` 추가를 줄이고, 반복되는 위젯은 `SListView`로 정리했습니다. 
위젯 수 자체가 성능에 직결되기 때문에, 계층 깊이도 줄이도록 노력했어요.

## UX 관련 피드백을 받아 개선한 경험이 있나요?
디자이너 분이 UI 이동 흐름이 너무 복잡하다는 피드백을 주셔서, 상태 머신을 단순화하고 탭 방식으로 변경했던 적이 있습니다. 
그 이후 접근성도 좋아지고, 클릭 수가 줄면서 반응도 더 좋아졌습니다. 
직접 사용하는 입장에서 피드백을 받는 게 항상 중요한 것 같아요.

🗝️ **키워드**: `Tab UI`, `State Simplification`, `User Feedback`, `Iteration`

# 언리얼 데디케이트 서버 연동 이해
## UI에서 서버 데이터를 어떻게 받아오나요?
보통 서버 데이터는 클라이언트의 **GameInstance**, **PlayerController**, 혹은 별도 **DataManager** 클래스에서 받아온 뒤, UI에서는 해당 데이터를 **Delegate로 구독하거나, Getter로 요청**받는 구조를 사용했습니다.  
데이터가 변경되었을 때는 **OnUpdated 이벤트**를 통해 UI 쪽에서 자동으로 갱신이 되도록 구성했어요.


## 서버에서 받은 퀘스트/이벤트 데이터를 UI로 갱신하는 구조는?
퀘스트 시스템의 경우, 서버에서 받은 원본 데이터를  `QuestManager` 같은 시스템에서 관리하고,  UI에는 UI 전용 가공 데이터 구조(FQuestViewData)로 전달하는 방식을 사용했습니다.  
퀘스트가 갱신되면 `OnQuestUpdated` 같은 Delegate가 Broadcast되고,  UI는 이를 구독해서 `Refresh()` 함수로 화면을 업데이트합니다.

🗝️ 키워드:  `OnQuestUpdated`, `Push 기반 UI 갱신`, `UI ViewModel`, `FQuestViewData`

## 데디케이트 서버와 클라이언트 UI 간 싱크 이슈가 있었을  때 어떻게 해결했나요?
실시간 상태나 퀘스트 완료 조건 등이 서버에서 먼저 처리되기 때문에, 간혹 클라이언트 UI와 서버 실제 상태가 맞지 않을 수 있어요. 
이런 경우엔 주로 **서버에서 최종 결과를 리턴받고**, UI는 그걸 기준으로 갱신하게끔 설계합니다.

🗝️ 키워드: `Authoritative Server`, `서버 응답 기준 갱신`, `예측 UI`, `보상 확정 타이밍`

# 기본 질문 리스트 (Mock Interview용)
- 본인이 직접 설계한 UI 시스템 중 기억에 남는 구조는?
    델리게이트를 활용한 구독-발행 방식과 인터페이스를 조합한 버튼 및 내비게이션 구조로서, 컨트롤러가 현재 조작하고있는 위젯에게 인터페이스를 통해 조작 이벤트를 전달하고 해당 위젯은 하위 위젯이 포커스 되어있다면 하위 위젯에게 먼저 전달하고 본인 로직을 처리하고 이를 bool 값으로 반환 받음으로서 단계적인 위젯 구성과 디버깅 혹은 확인을 동시에 체크할 수 있게하였습니다.
    
    정리한 대답
    `UUserWidget` 기반으로 `INavigationReceiver` 인터페이스를 설계하여 입력 이벤트를 위임하고, `FInputDelegate`를 사용해 각 UI가 원하는 타이밍에 바인딩 해제/등록할 수 있게 했습니다. 이렇게 해서 계층적인 UI 내에서 Top-Down 입력 처리를 안정적으로 수행했어요.
    
- Blueprint와 C++ 혼합 개발 시 기준은 어떻게 잡나요?
    기본 뼈대는 C++ 로 구성하되 자주 바뀌는 변수값 혹은 직접 보며 변경하는 부분들은 블루프린트에서 바로 변경 가능하도록 UPROPERTY, UFUNCTION 매크로를 붙여 처리합니다.
    또한 복잡한 로직은 주로 C++ 함수에서 처리하고 블루프린트는 이를 활용(호출)하는 방식으로 처리합니다.
    
    추가 대답
    예를 들어 `SkillName`, `IconTexture`, `CooldownSeconds` 같은 값들은 디자이너가 직접 수정하니 Blueprint에서 처리할 수 있게 UPROPERTY로 노출시킵니다.
    
- 드래그 앤 드롭을 UMG에서 처리하는 방식은?
    저라면 별도의 커스텀 버튼을 구축하여 Pressed 상태를 확인 받으며 유지되는 동안 화면상의 마우스의 좌표값을 업데이트 받도록합니다. 
    이후 다른 공간의 위젯이 겹쳐질때 해당 위젯의 데이터를 콜백함수로 반환받아 저장하고 Release 된다면 겹쳐진 위젯이 있다면 별도의 처리 함수를 구현하고, 없다면 그저 그자리에 위치(혹은 오프셋 처리된 위치)로 설정할 것 같습니다.
    
    정리한 대답
    "기본적으로 `OnDragDetected`에서 DragOperation을 생성하고, `OnDrop`에서 대상 위젯의 처리를 분기합니다. 다만 복잡한 UI에서는 커스텀 드래그 매니저를 만들어서 직접 마우스 좌표를 트래킹하는 방식도 사용해봤습니다."
    
- Slate로 커스텀 위젯 만든 경험이 있다면 SCompoundWidget을 어떻게 활용했는지?
    SCompoundWidget 은 잘모르겠네요.
    
    정리한 대답
    "아직 `SCompoundWidget`은 직접 활용해보진 않았지만, Slate로 툴 개발을 하거나 커스텀 위젯이 필요한 경우에 사용되는 건 알고 있고, 구조와 사용법은 학습 중입니다."
    
- UI 성능 튜닝 관련 도구나 방법은?
    stat gpu 커맨드를 통해 현재 Slate UI 가 어느정도의 자원을 소모하는지 알 수 있으며, 언리얼에서 제공하는 인사이트 툴을 통해 어디서 어떻게 얼만큼 자원을 쓰는지 알 수 있는걸로 알고있습니다.
    
    정리한 대답
    "Slate 성능 확인은 `Stat Slate`, `Stat GPU`로 하고, UMG에선 `Invalidation Box`로 리렌더링 최소화, `Retainer Box`로 복잡한 위젯을 캐시해 처리합니다."
    
- 콘텐츠 시스템과 UI 간 데이터 전달은 어떤 식으로 구성했는지?
    주로 콘텐츠 시스템에서 UI용으로 가공된 데이터를 Get 함수로 준비하여두고 UI는 콘텐츠 시스템 델리게이트 바인딩하여, 변경점(Broadcast)이 생길때 이를 요청하여 UI를 업데이트 하는 방식을 위주로 사용합니다.
    
    보완점
    DataAsset, Struct 등 활용한 적 있다면 추가해도 좋습니다.
    
- 액션 게임 특화 UI 설계 시 어떤 점을 고려하나요? (ex: 빠른 반응, 애니메이션, HUD 구조)
    시각적 표현 위주로 구성하는데 중점을 두지 않을 까 싶습니다. 적 캐릭터가 얼마나 많이 데미지를 받았는지 이를 시각적으로 표현하거나(줄어들 게이지 양의 색깔을 바꿈 혹은 번쩍임), 또는 유저가 전투 상황에서 쉽고 직관적이게 상태를 파악할 수 있는게 중요하다고 생각합니다.
    
    보완점
    “애니메이션 최적화”, “즉각적인 피드백”, “시야를 방해하지 않는 위치” 등 키워드를 덧붙이면 더 고급스럽게 느껴져요.
    
- 데디케이트 서버 기반 프로젝트에서 UI 설계 시 신경써야 할 부분은?
	보내는 패킷의 양이 적어야 하므로 실제 데이터값을 전부 보내는게 아닌 UI에 필요한 요소들을 최적화해서 보내는게 제일 중요할 것 같습니다.
	또한 실제 데이터와 UI 상 싱크 이슈가 있으면 안되겠죠.
	
	정리한 대답
	"데디케이트 서버 구조에서는 UI는 클라이언트 전용이므로, Replication은 UI에 필요한 최소 데이터만 대상으로 하고, UI는 이를 이벤트로 받아 업데이트하도록 구성합니다."


# 주요 질문
Q. Slate로 커스텀 위젯 만들 때 주의할 점?
A. 공유 포인터 기반 구조로 인해 순환 참조 방지를 위해 WeakPtr 사용이 중요하며, 시각적 피드백이 어렵기 때문에 위젯 계층을 분리하고 구조적으로 관리해야 합니다.

Q. 게임 UI와 툴 UI 차이는?
A. 게임 UI는 Viewport 기반 HUD 구조이고, 툴 UI는 SDockTab을 통해 독립적인 창을 띄워 사용합니다. Slate는 월드와 무관한 UI 제작에 적합합니다.

Q. UI 모듈화는 어떻게 하나요?
A. 플러그인 형태로 제작하여 프로젝트 간 재사용하며, 위젯, 스타일, 데이터 구조를 분리해 유지보수성을 높입니다.

Q. UMG에서 Drag & Drop 구현 핵심은?
A. UDragDropOperation을 상속해 커스텀 로직을 구성하고, OnDragDetected, OnDrop 이벤트를 활용해 시각 피드백과 로직 처리를 분리합니다.

Q. UMG와 Slate 혼합 경험은?
A. 직접 혼합은 없지만, Slate 플러그인 구현 경험이 있고, UMG 위젯을 Slate에 포함시키는 구조도 이해하고 있어 대응 가능하다고 생각합니다.

Q. UI 관련 성능 이슈와 해결 사례는?
A. 고해상도 텍스처 렌더링에서 Slate 렌더 타임 이슈 발생 → RetainerBox, InvalidationBox로 캐싱 처리, Unreal Insights 및 stat GPU 활용해 병목 제거
